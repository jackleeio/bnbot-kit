# /schedule — X post scheduler (macOS launchd)

## Why launchd, not CronCreate

`CronCreate` is **session-only by default** — when the Claude REPL exits
(closed, crashed, ended mid-turn), the schedule is lost. Even `durable:
true` needs the REPL running at fire time.

**launchd** is macOS's native scheduler:
- Runs whenever the user is logged in, Chrome or no Chrome, bnbot or no bnbot
- Survives reboots + closed Claude sessions
- Handles missed one-shot firings via catch-up on next login
- Logs to disk at a known path

This is the only approach that delivers the subscription promise of
"scheduled posts fire while I sleep".

## Two modes — pick based on intent

| Intent shape | Mode | Storage |
|---|---|---|
| **Calendar** — bulk plans, daily brief queue, inspectable | `bnbot calendar add` + one long-lived tick agent | `~/.bnbot/calendar/<YYYY-MM-DD>.json` |
| **Ad-hoc** — "5 分钟后发 X", one-shot fire-and-forget | per-task launchd plist | `~/Library/LaunchAgents/com.bnbot.schedule.<id>.plist` |

**Default to calendar mode** unless the user wants a true one-shot.
Calendar wins for:
- Multi-day content planning (weekly brief → 20 entries across 7 days)
- "排到明早 9 点发 X" (append to tomorrow's file, no new plist)
- Batch operations (cancel/reschedule = edit JSON, not bootout N plists)
- Desktop UI Calendar tab reads/writes the same JSON directly

Calendar needs a one-time install: `bnbot calendar install` — registers
`com.bnbot.calendar-tick` launchd agent that wakes every 300s, reads
today + yesterday's JSON, fires pending entries whose local time has
arrived (1h catch-up for laptop sleep).

## Calendar mode — how to use

### First-time setup (once per machine)

```bash
bnbot calendar install
```

Check it's loaded:

```bash
launchctl list | grep com.bnbot.calendar-tick
```

### Add an entry

```bash
bnbot calendar add \
  'bnbot x post --engine debugger "morning"' \
  --at 09:00 --date 2026-04-22 --note 'morning greeting'
```

Returns the entry's id. Omit `--date` to schedule for today.

### Batch-add (weekly brief → N entries)

```bash
bnbot calendar add 'bnbot x post --engine debugger "..."' --at 09:00 --date 2026-04-22
bnbot calendar add 'bnbot x post --engine debugger "..."' --at 13:30 --date 2026-04-22
bnbot calendar add 'bnbot x thread --engine debugger "[...json...]"' --at 19:00 --date 2026-04-23
```

### Inspect

```bash
bnbot calendar list                    # today
bnbot calendar list --date 2026-04-22
bnbot calendar list --all              # summary of every day file
```

### Cancel / reschedule

```bash
bnbot calendar remove <id>             # finds across all days
# reschedule = remove + add
```

### How it fires

The tick agent wakes every 5 min and for each pending entry where
`now ≥ scheduled_time` AND `now - scheduled_time ≤ 1h`:
- mark `status: running`
- exec the `action` shell command
- on success: mark `status: done`, stash stdout into `lastResult`
- on failure: increment `attempts`; retry next tick up to 3 times
- if past the 1h catch-up window: mark `status: failed` with
  "missed firing window"

Log: `~/.bnbot/logs/calendar-tick.log` (append-only, stdout+stderr of
the tick process).

## Ad-hoc mode (single plist) — for one-shot "in N minutes"

Paths:

```
~/Library/LaunchAgents/com.bnbot.schedule.<id>.plist   # job definition
~/.bnbot/bin/fire-<id>.sh                              # wrapper script
~/.bnbot/logs/schedule-<id>.log                        # stdout + stderr
```

Always `mkdir -p ~/.bnbot/bin ~/.bnbot/logs` before writing.

### Creating a one-shot schedule

User says "明天早上 8 点发 'morning!'":

### 1. Compute fire time + generate id

```bash
ID=$(uuidgen | tr '[:upper:]' '[:lower:]' | cut -c1-8)
# Target time in user's local TZ. Use `date` math or explicit Y/M/D/H/M.
```

### 2. Write the wrapper script

```bash
# Resolve bnbot's absolute path RIGHT NOW. If not in PATH at schedule
# creation, abort — don't write a wrapper that silently fails at fire
# time. Avoids hardcoding /opt/homebrew vs /usr/local per machine.
BNBOT_BIN="$(command -v bnbot)"
if [ -z "$BNBOT_BIN" ]; then
  echo "ERROR: bnbot CLI not found in PATH. Install first: npm install -g @bnbot/cli" >&2
  exit 1
fi
BNBOT_DIR="$(dirname "$BNBOT_BIN")"

# Heredoc is single-quoted so the wrapper body uses literal $HOME /
# $(date) etc that evaluate at fire time, not at creation time.
# Placeholders (__BNBOT__, __ID__, __ACTION__) get sed-substituted after.
cat > ~/.bnbot/bin/fire-$ID.sh <<'WRAPPER'
#!/bin/bash
# Auto-generated by /schedule skill. Fires one scheduled X post.
set -o pipefail
exec >> "$HOME/.bnbot/logs/schedule-__ID__.log" 2>&1

# Restore a sensible PATH — launchd strips it to /usr/bin:/bin only.
# The bnbot bin's own dir goes first so we always find the right copy.
export PATH="__BNBOT_DIR__:/opt/homebrew/bin:/usr/local/bin:$HOME/.local/bin:/usr/bin:/bin:/usr/sbin:/sbin"
BNBOT="__BNBOT__"

echo "[$(date '+%Y-%m-%d %H:%M:%S')] fire-__ID__ start (bnbot=$BNBOT)"

# 1. Chrome must be running (bnbot extension + CDP needs it)
if ! pgrep -f "Google Chrome" > /dev/null; then
  echo "[$(date '+%H:%M:%S')] Chrome not running → open -a"
  open -a "Google Chrome"
  sleep 5
fi

# 2. bnbot serve daemon auto-spawns on first CLI call; give it time to
#    handshake with the extension WS.
sleep 3

# 3. Fire the action with a retry loop. $BNBOT instead of bare `bnbot`
#    because launchd's stripped PATH would miss it even with our export.
for attempt in 1 2 3; do
  if __ACTION__ ; then
    echo "[$(date '+%H:%M:%S')] success (attempt $attempt)"
    # One-shot self-cleanup.
    launchctl bootout gui/$UID "$HOME/Library/LaunchAgents/com.bnbot.schedule.__ID__.plist" 2>/dev/null || true
    rm -f "$HOME/Library/LaunchAgents/com.bnbot.schedule.__ID__.plist"
    rm -f "$HOME/.bnbot/bin/fire-__ID__.sh"
    exit 0
  fi
  echo "[$(date '+%H:%M:%S')] attempt $attempt failed, retry in 30s"
  sleep 30
done

echo "[$(date '+%H:%M:%S')] all retries exhausted, giving up"
exit 1
WRAPPER
chmod +x ~/.bnbot/bin/fire-$ID.sh

# Substitute placeholders after the heredoc so the wrapper stays literal.
# The action MUST use $BNBOT (not bare bnbot) — see ## Action templates.
ACTION='$BNBOT x post --engine debugger "morning!"'
sed -i '' \
  -e "s|__ID__|$ID|g" \
  -e "s|__BNBOT_DIR__|$BNBOT_DIR|g" \
  -e "s|__BNBOT__|$BNBOT_BIN|g" \
  -e "s|__ACTION__|$ACTION|g" \
  ~/.bnbot/bin/fire-$ID.sh
```

> The wrapper above is for **one-shot**. For recurring, remove the
> self-cleanup block (leave the plist in place so it fires again).

### 3. Write the launchd plist

For one-shot at a specific date/time (e.g. 2026-04-21 08:00):

```bash
cat > ~/Library/LaunchAgents/com.bnbot.schedule.$ID.plist <<PLIST
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.bnbot.schedule.$ID</string>
  <key>ProgramArguments</key>
  <array>
    <string>/bin/bash</string>
    <string>$HOME/.bnbot/bin/fire-$ID.sh</string>
  </array>
  <key>StartCalendarInterval</key>
  <dict>
    <key>Month</key><integer>4</integer>
    <key>Day</key><integer>21</integer>
    <key>Hour</key><integer>8</integer>
    <key>Minute</key><integer>0</integer>
  </dict>
  <key>StandardOutPath</key>
  <string>$HOME/.bnbot/logs/schedule-$ID.log</string>
  <key>StandardErrorPath</key>
  <string>$HOME/.bnbot/logs/schedule-$ID.log</string>
  <key>RunAtLoad</key>
  <false/>
</dict>
</plist>
PLIST
```

Recurring daily at 9am:

```xml
<key>StartCalendarInterval</key>
<dict>
  <key>Hour</key><integer>9</integer>
  <key>Minute</key><integer>0</integer>
</dict>
```

Recurring every Monday at noon (Weekday: 0=Sun, 1=Mon, ... 6=Sat):

```xml
<key>StartCalendarInterval</key>
<dict>
  <key>Weekday</key><integer>1</integer>
  <key>Hour</key><integer>12</integer>
  <key>Minute</key><integer>0</integer>
</dict>
```

### 4. Load the plist

```bash
launchctl bootstrap gui/$UID ~/Library/LaunchAgents/com.bnbot.schedule.$ID.plist
```

Verify:
```bash
launchctl list | grep com.bnbot.schedule.$ID
```

Confirm to user:
> 已排期：2026-04-21 08:00 发『morning!』（task id: `<id>`）
> 日志：`~/.bnbot/logs/schedule-<id>.log`

## Listing schedules

```bash
ls ~/Library/LaunchAgents/com.bnbot.schedule.*.plist 2>/dev/null | while read f; do
  id=$(basename "$f" | sed 's/^com.bnbot.schedule.//;s/\.plist$//')
  echo "--- $id ---"
  grep -A1 "Hour\|Minute\|Day\|Month\|Weekday" "$f" | grep integer
  grep -E "launchctl|exit 0" ~/.bnbot/logs/schedule-$id.log | tail -3
done
```

Format for user:

```
📅 已排期
  <id-1>  明早 08:00  发『morning!』                   [pending]
  <id-2>  每天 09:00  /bnbot 早安                      [recurring]
  <id-3>  每周一 12:00 /bnbot 周一复盘                 [recurring]
```

## Cancelling

```bash
launchctl bootout gui/$UID ~/Library/LaunchAgents/com.bnbot.schedule.$ID.plist
rm ~/Library/LaunchAgents/com.bnbot.schedule.$ID.plist
rm -f ~/.bnbot/bin/fire-$ID.sh
```

Don't delete the log file automatically — user might want to review
why the task succeeded/failed.

## Viewing logs

```bash
tail -f ~/.bnbot/logs/schedule-<id>.log   # live tail
cat  ~/.bnbot/logs/schedule-<id>.log      # full history
```

## Action templates

Inside the wrapper's `__ACTION__`, use `$BNBOT` (the variable exported
by the wrapper) instead of bare `bnbot` — launchd's PATH may not find
the unqualified name. Always use `--engine debugger` (never DOM).

```bash
# Simple post
$BNBOT x post --engine debugger "text"

# Post with media (use absolute paths that exist at fire time)
$BNBOT x post --engine debugger "text" --media /Users/jacklee/Pictures/today.png

# Thread
$BNBOT x thread --engine debugger '[{"text":"1/3"},{"text":"2/3"},{"text":"3/3"}]'

# Reply / quote
$BNBOT x reply  --engine debugger "https://x.com/.../status/..." "text"
$BNBOT x quote  --engine debugger "https://x.com/.../status/..." "text"
```

### Dynamic content (Claude-drafted at fire time)

Bake a call to `bnbot` (the Claude Code fork) into the action, so a fresh
Claude session composes the tweet at fire time. Note: this requires
bnbot CLI to be available on PATH and has a cold-start cost.

```bash
cd ~/Projects/bnbot && bun run src/entrypoints/cli.tsx -p \
  "生成一条早安推文（用户 voice），然后用 CDP 直接发布" \
  --model=sonnet
```

Prefer static strings for simple cases. Only use `-p` invocation when
the content must be fresh.

## Escaping rules

The action string ends up inside a bash heredoc inside a sed command.
Tricky characters to handle:

- Double quotes inside the tweet: escape as `\"` in the heredoc, or
  switch the whole action to single quotes and escape single quotes
  inside as `'\''`.
- `$` in the text: escape as `\$` to prevent shell expansion.
- Backticks: escape as `\\\``.
- Emojis: safe (Bun + bash handle UTF-8 fine).

Simpler strategy: write the tweet text to a temp file, then have the
wrapper `cat` it:

```bash
echo '复杂"包含$符号\的推文' > ~/.bnbot/data/tweet-$ID.txt
# wrapper action becomes:
# bnbot x post --engine debugger "$(cat ~/.bnbot/data/tweet-__ID__.txt)"
```

## Don't do

- Don't use `crontab` — user likely has other cron jobs and we should
  not touch their crontab. launchd is per-job and isolated.
- Don't default to session-only (`CronCreate`). That was the old
  approach — it doesn't survive the REPL closing.
- Don't start a long-running daemon just for scheduling. launchd IS
  the daemon.
- Don't forget `mkdir -p ~/.bnbot/bin ~/.bnbot/logs` before writing
  files.
- Don't skip `chmod +x` on the wrapper — launchd requires executable.

## Time zone

launchd runs in the system's local timezone. User's phrasing ("8am")
means local. No UTC conversion needed.

## Debug sequence when a schedule doesn't fire

1. `launchctl list | grep com.bnbot.schedule.<id>` — is it loaded?
2. `cat ~/.bnbot/logs/schedule-<id>.log` — did it run but fail?
3. `plutil -p ~/Library/LaunchAgents/com.bnbot.schedule.<id>.plist` —
   validate the plist structure + verify the time fields
4. `launchctl start com.bnbot.schedule.<id>` — fire immediately to test
   the wrapper (bypasses the schedule)

## Common failure mode: `bnbot: command not found`

launchd runs with `PATH=/usr/bin:/bin:/usr/sbin:/sbin` — it does NOT
inherit the user's interactive shell PATH. `bnbot` (installed to
`/opt/homebrew/bin` on Apple Silicon, `/usr/local/bin` on Intel or with
npm global) will NOT be found unless the wrapper exports PATH first.

The wrapper template above already handles this — keep the
`export PATH=...` line. If someone disables it, every schedule will
silently fail its 3 retries and exit 1 with "command not found" errors
in the log.
